InnerCircle

A Small-Circle Social Media Platform
Qiwen Lin  ·  Irys Zhang  ·  Weijie Zhu  ·  Zhengyang Li
University of Toronto — CSC309 Winter 2026

The Problem

Mainstream social media encourages quantity over quality

  • Platforms like Instagram, Twitter, Facebook prioritize large follower counts and engagement metrics
  • Users experience shallow interactions, social comparison, and content overload
  • Research shows people maintain meaningful relationships with only 5–15 close friends (Dunbar's Number)

Existing solutions fall short

PlatformLimitation
BeRealStill open follower model, no cap
IG Close FriendsAn afterthought, not core design
WhatsApp GroupsNo social feed, profiles, discovery
None of these structurally enforce small, meaningful networks.

Our Solution: InnerCircle

A social platform that limits your circle to ~20–30 friends

Core Concept

  • Enforced friend cap — must remove a friend to add a new one at the limit
  • Every interaction happens within a small, trusted group
  • Social media that feels personal, not performative

Target Users

  • Young adults and students
  • Want to share with close friends through photo sharing and real-time messaging
  • Without the noise of large-scale social networks

Key Differentiators

1. Finite Social Graph — Friend cap is a structural constraint, not just a config flag. Enforced at database transaction level.

2. Friend-Based Access Control — Every post, message, and notification is gated by friendship status.

3. Real-Time First — Chat, notifications, online/offline status all delivered via WebSockets.

Core: Authentication & Authorization

Better Auth with session-based authentication + centralized access control

Authentication Flow

Register / Login → Better Auth validates → Session created → Cookie set

Implementation

  • Session-based auth (7-day expiry, 1-day refresh)
  • Password minimum 8 characters
  • Middleware: unauthenticated → /login
  • Middleware: authenticated on auth pages → /feed
POST /api/auth/sign-up    → Register
POST /api/auth/sign-in    → Login
GET  /api/auth/session     → Current session
POST /api/auth/sign-out    → Logout

Centralized Auth Utilities

// src/lib/auth-utils.ts

// For Server Components (redirects)
async function requireSession()

// For API routes (throws 401)
async function requireSessionForApi()

// General session check
async function getSession()
Security Pattern: Every API route starts with requireSessionForApi(). One file to audit — src/lib/auth-utils.ts.

Route Protection

  • Middleware intercepts all non-public, non-API routes
  • Session cookie checked via getSessionCookie()
  • Public paths: /, /login, /register

Core: Friend System with Cap Enforcement

The defining feature — structural friend limit enforced at database level

Friend Request Flow

Search user → Check cap → Send request → Addressee checks cap → Accept / Decline

Cap Enforcement

  • Default cap: 25 friends per user
  • Cap checked on both sending AND accepting
  • At cap → must remove a friend before adding new one
  • UI shows progress bar: "12 / 25 friends"

Race Condition Protection

// Atomic transaction prevents concurrent 
// accepts from exceeding the cap
await prisma.$transaction(async (tx) => {
  const count = await tx.friendship.count({...})
  if (count >= user.friendCapLimit) throw Error
  await tx.friendship.update({status: "ACCEPTED"})
})

Socket.io — One-on-one messaging with typing indicators and read receipts

Architecture

Socket Events

EventDirectionPurpose
chat:message↔Send/receive message
chat:typing→Typing indicator
chat:read→Mark as read
user:online←User connected
user:offline←User disconnected

Message Persistence

  • Messages saved to PostgreSQL via Prisma before being emitted
  • Conversation model for efficient listing and unread counts
  • readAt timestamp for read receipt tracking
  • Message history loaded from API, new messages via Socket

Client Integration

  • SocketProvider connects on login, disconnects on logout
  • Incoming events dispatched to Redux slices
  • Instant UI updates across all components
Design Decision: Custom HTTP server chosen from Day 1 (not a fallback) — Next.js API Routes cannot handle WebSocket upgrades. This eliminates the integration risk entirely.

Advanced Feature 2: Image Processing Pipeline

Server-side compression, WebP conversion, and thumbnail generation with Sharp

Pipeline Flow

Client Upload (≤10MB) → Validate type & size → Sharp: resize + WebP → Sharp: thumbnail → Upload to S3 → Return URLs

Post Image Processing

// Full size: max 1920px, WebP @ 80%
sharp(input)
  .resize(1920, 1920, { fit: "inside" })
  .webp({ quality: 80 })
  .toBuffer()

// Thumbnail: 400x400 crop, WebP @ 60%
sharp(input)
  .resize(400, 400, { fit: "cover" })
  .webp({ quality: 60 })
  .toBuffer()

Avatar Processing

// 256x256 crop, WebP @ 80%
sharp(input)
  .resize(256, 256, { fit: "cover" })
  .webp({ quality: 80 })
  .toBuffer()

Storage

  • AWS SDK v3 for S3-compatible storage (DigitalOcean Spaces)
  • Organized folders: posts/, posts/thumbnails/, avatars/
  • UUID-based filenames prevent collisions
  • Public-read ACL for direct URL access
Design Decision: Compress before upload to S3 (not after). Client-side 10MB limit keeps Sharp's work manageable. No background job queue needed.

Advanced Features 3 & 4

AI Content Moderation (External API)

Non-blocking, fail-open design

Upload succeeds → Async moderation → Flag if unsafe
  • Image uploads succeed immediately
  • Moderation runs in the background
  • 10-second timeout prevents hanging
  • API down → fail open (allow image)
  • Flagged images logged for review
// Fire-and-forget pattern
moderateImage(url).then((result) => {
  if (!result.safe)
    console.warn("Flagged:", result.categories)
})
Satisfies External API Integration requirement.

Redux Toolkit State Management

Global state for real-time data

SlicePurpose
feedSlicePosts, pagination, optimistic likes
chatSliceConversations, messages, typing
notificationSliceAlerts, unread count
friendSliceFriends, requests, online status

Why Redux over React Context?

  • Multiple real-time data sources (WebSocket + API + optimistic updates)
  • createAsyncThunk for API calls with loading/error
  • Socket.io events dispatch directly → instant UI updates
  • No race conditions or stale data
Satisfies Advanced State Management requirement.
